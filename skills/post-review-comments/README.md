# post-review-comments

Triage findings from a `panel-review` (or any structured review with `file:line` references) and post the approved ones as inline comments on a GitHub PR. Walks each finding one at a time, lets you tweak the wording, and batches everything into a single PR review at the end.

## Install

```
npx skills add catena-labs/skills --skill post-review-comments
```

## How to use it

After running `panel-review` (or any review that produced findings with `file:line` refs), just ask:

- "post these comments to the PR"
- "let me triage and post these"
- "turn these findings into PR comments"
- "comment these on PR 85"
- "post the panel-review findings to GitHub"

The skill walks each finding one at a time and asks what to do — post as-is, soften the tone, make more concise, skip, or rewrite however you want. Approved comments are batched into one PR review at the end so you get one notification, not twenty.

## What it does

- Resolves the PR via `gh pr view` (you pass the PR ref explicitly — it does not auto-detect from the current branch).
- For each finding: shows the raw panelist output, asks how to handle it, optionally rewrites the body to match the tone you want, and stages it.
- Batches every approved finding into a single `POST /pulls/{pr}/reviews` call so all inline comments land together.
- Appends a small footer to every comment (`generated by <panelist> via panel-review`) so the provenance stays visible.
- Reports any comments GitHub rejected (e.g., line not in the PR diff) so you can post them manually instead of having them silently dropped.

## Gotchas

- **GitHub only allows inline comments on lines inside the PR diff hunks.** Findings that point at unchanged context outside a hunk get rejected by the API — the skill surfaces these so you can fall back to a regular issue comment.
- **No PR auto-detection.** Branch state can drift between running the review and posting comments. The skill takes the PR ref from the calling assistant's context (which already knows it from the review's scope) rather than guessing.
- **One review, not many.** Posting comments individually scatters them across the PR timeline and spams notifications. The skill always batches.
- **The footer link is generic.** It points at the skills repo, not at a per-run artifact — `panel-review` does not post a tracking comment, so there is nothing per-run to link to.
